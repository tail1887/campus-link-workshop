# 01. Product Planning

## 1) 프로젝트 한 줄 소개

- 프로젝트명: `Campus Link`
- 한 줄 설명: 캠퍼스 안에서 스터디와 프로젝트 팀원을 빠르게 찾고 지원할 수 있는 웹 기반 매칭 플랫폼
- 개발 기간: 워크숍 MVP 기준 1주
- 팀 구성: 프론트엔드 중심 3~4명 협업을 가정

## 2) 문제 정의

현재 대학생은 스터디나 프로젝트 팀원을 구할 때 여러 커뮤니티와 채팅방을 오가며 정보를 흩어지게 관리하는 경우가 많다.

- 문제 1: 모집 글이 여러 채널에 흩어져 있어 비교와 탐색이 어렵다.
- 문제 2: 어떤 팀이 어떤 목표와 분위기를 원하는지 한 화면에서 파악하기 어렵다.
- 기존 방식의 한계: 게시글 작성, 탐색, 지원 흐름이 분리되어 있어 발표용 데모로 보여주기에도 임팩트가 약하다.

## 3) 타겟 사용자

- 핵심 사용자: 스터디, 사이드 프로젝트, 해커톤 팀원을 구하거나 찾는 대학생
- 사용 상황: 학기 초 팀 구성, 공모전 준비, 포트폴리오 프로젝트 모집, 시험 기간 스터디 결성
- 사용 빈도/맥락: 단기간 집중 탐색 후 빠르게 의사결정을 내리는 상황

## 4) 목표

- 제품 목표: 모집 글 탐색부터 지원까지 이어지는 흐름을 하나의 서비스로 보여준다.
- 사용자 경험 목표: 메인 화면만 봐도 서비스 가치가 전달되고, 목록-상세-지원 흐름이 직관적으로 이어진다.
- 기술 목표: Vercel에 즉시 배포 가능한 Next.js 기반 MVP를 완성하고, 서버 없이도 데모 가능한 mock 흐름을 제공한다.

## 5) MVP 범위

이번 프로젝트에서 반드시 구현할 것:

- [ ] 메인 랜딩 페이지
- [ ] 모집글 목록 조회
- [ ] 모집글 상세 조회
- [ ] 글쓰기 폼과 mock 저장
- [ ] 지원하기 mock 제출
- [ ] 최소 검증과 배포 가능한 실행 환경

기능별 상세:

| 기능 | 설명 | 우선순위 | 포함 여부 |
| --- | --- | --- | --- |
| 메인 페이지 | 서비스 가치, 카테고리, 추천 모집글, CTA를 강조하는 랜딩 화면 | High | Yes |
| 모집글 목록 | 카드형 리스트, 카테고리 필터, 검색, 상태 배지 제공 | High | Yes |
| 모집글 상세 | 팀 소개, 목표, 기술 스택, 모집 역할, 일정, 현재 지원 수와 지원 CTA 제공. 작성자는 로컬 demo 기준으로 마감/삭제 관리 가능 | High | Yes |
| 글쓰기 | 제목, 카테고리, 역할, 인원, 일정, 설명을 입력하고 mock 생성. 긴 텍스트는 UI를 넘기지 않도록 자동 줄바꿈과 글자수 제한 제공 | High | Yes |
| 지원하기 mock | 지원 메시지와 연락처를 입력하고 성공 피드백을 확인 | High | Yes |
| 대시보드/마이페이지 | 내가 쓴 글과 지원 현황 관리 | Medium | No |
| 실시간 채팅 | 지원 후 즉시 대화 | Low | No |

## 6) 비MVP 범위

이번 프로젝트에서 의도적으로 제외할 것:

- [ ] 실제 회원가입 / 로그인
- [ ] 실시간 알림 및 채팅
- [ ] 관리자 승인 / 신고 기능
- [ ] 실제 DB 연동과 영구 저장
- [ ] 추천 알고리즘

## 7) 핵심 사용자 흐름

가장 중요한 사용자 흐름 3개:

### Flow A. 메인에서 모집글 탐색까지

1. 사용자는 메인 페이지에서 현재 인기 카테고리와 추천 모집글을 본다.
2. 시스템은 주요 지표와 큐레이션 카드를 보여주며 서비스 가치를 설명한다.
3. 사용자는 목록 페이지로 이동해 검색과 필터를 적용한다.
4. 결과적으로 자신에게 맞는 모집글 후보를 빠르게 좁힌다.

### Flow B. 모집글 상세 확인 후 지원하기

1. 사용자는 목록에서 관심 있는 모집글 카드를 선택한다.
2. 시스템은 모집 목표, 팀 구성, 필요 역할, 마감 일정 정보를 상세 화면에 제공한다.
3. 사용자는 지원하기 mock 폼에 자기소개와 연락처를 입력한다.
4. 성공 시 지원 완료 메시지와 다음 액션 안내를 확인한다.

### Flow C. 새 모집글 작성

1. 사용자는 글쓰기 페이지에서 모집 유형, 제목, 설명, 역할, 인원 정보를 입력한다.
2. 시스템은 필수 값을 검증하고 mock 게시글을 생성한다.
3. 사용자는 생성 직후 상세 화면으로 이동한다.
4. 결과적으로 발표 시 작성부터 상세 노출까지 한 번에 시연할 수 있다.

## 8) 성공 기준

- [ ] 메인, 목록, 상세, 글쓰기, 지원하기 흐름이 데모에서 끊기지 않는다.
- [ ] 모집글 작성과 지원하기가 최소한의 입력 검증을 제공한다.
- [ ] Vercel에 배포 가능한 구조를 갖춘다.
- [ ] README와 실제 화면 구성이 일치한다.

## 9) 가정 및 오픈 질문

이번 초안은 아래를 가정한다.

- [ ] 인증 없이 누구나 탐색하고 mock 행동을 수행한다.
- [ ] 데이터는 seed 데이터와 브라우저 localStorage를 함께 사용한다.
- [ ] 발표용 임팩트를 위해 UI 완성도를 기능 범위보다 우선한다.

추후 팀에서 확인할 것:

- [ ] 실제 서비스 단계에서 학교 인증을 붙일지 여부
- [ ] 실제 DB와 백엔드 API를 어떤 시점에 붙일지 여부
- [ ] 모바일 우선 발표인지 데스크톱 우선 발표인지 여부

## 10) Post-MVP 확장 로드맵

MVP 이후에는 아래 확장 흐름을 큰 제품 방향으로 본다.

### Phase 1. 계정과 온보딩 기반

- 로그인 / 회원가입
- 회원가입 설문과 관심 키워드 선택
- 사용자 기본 프로필
- 관리자 기본 프로필과 역할 구분

이 단계를 먼저 두는 이유:

- 이후의 프로필 개인화, 알림, 관리자 권한, AI 보조 기능이 모두 사용자 식별과 역할 구분에 의존하기 때문이다.

### Phase 2. 프로필 신뢰도와 커뮤니케이션 기반

- 추가 인증과 인증 마크
- 이력서 기본 편집 흐름
- 문의하기 기능
- 알림 설정

이 단계를 Phase 1 다음에 두는 이유:

- AI 자동 완성이나 운영 기능보다 먼저, 사용자의 신뢰도와 연락 경로, 입력 데이터의 기본 구조가 고정되어야 하기 때문이다.

### Phase 3. AI 보조 사용자 생산성

- GitHub 등록
- AI 프로젝트 분석
- 이력서 AI 자동 완성
- 모집글 작성 자동 완성

이 단계를 Phase 2 다음에 두는 이유:

- AI 기능은 계정, 프로필, 이력서, GitHub 연결 정보 같은 선행 입력이 갖춰져야 품질과 재사용성이 나온다.

### Phase 4. 관리자 운영과 자동화

- 게시글 관리
- AI 자동 필터링
- AI 필터 강도 조절
- 문의 / 알림 운영 처리

이 단계를 마지막에 두는 이유:

- 관리자 기능은 사용자/콘텐츠/알림 이벤트가 실제로 쌓여야 가치가 커지고, AI 운영 기능도 앞선 페이즈의 데이터 계약을 재사용하는 편이 안전하다.

확장 로드맵 운영 원칙:

- 한 번에 전체를 구현하지 않고, Phase 단위로 완료 기준을 넘긴 뒤 다음 Phase로 이동한다.
- 각 Phase는 4명이 동시에 움직일 수 있도록 병렬 작업으로 쪼갠다.
- 구체적인 의존성 트리와 브랜치 분할은 `docs/05-codex-collaboration-playbook.md`를 따른다.
